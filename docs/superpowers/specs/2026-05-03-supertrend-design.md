# supertrend — R package design

**Date:** 2026-05-03
**Status:** Approved (brainstorming complete; ready for implementation planning)

## Goal

A CRAN-quality R package providing the SuperTrend technical indicator and its trading signals, designed to plug directly into the `quantmod` charting workflow.

## Scope

In scope (v1):

- Classic SuperTrend calculation (single line, ATR-based, configurable ATR smoothing method)
- Trading signal extraction (entry/exit events on trend flips)
- quantmod-idiomatic chart overlay (`addSuperTrend()` / `chartSuperTrend()`)
- CRAN-readiness: roxygen2 docs, testthat tests, one vignette, `R CMD check --as-cran` clean

Out of scope (deferred):

- Multi-timeframe / multi-line "triple SuperTrend" overlays
- Alternative band methods (median price instead of hl2, etc.)
- Standalone ggplot2 visualization
- Rcpp performance optimization (only if profiling warrants it)
- GitHub Actions CI, `pkgdown` site

## Public API

Four exported functions:

| Function | Purpose |
|---|---|
| `SuperTrend(HLC, n = 10, multiplier = 3, atr_method = "wilder")` | Compute the indicator. Returns an xts with columns `supertrend`, `trend` (+1/−1), `upper_band`, `lower_band`. |
| `supertrend_signals(st)` | Take a `SuperTrend()` output, return an xts with columns `long_entry`, `short_entry`, `long_exit`, `short_exit` (0/1 integer flags on flip bars). |
| `addSuperTrend(n = 10, multiplier = 3, atr_method = "wilder", col = c("#26a69a", "#ef5350"), lwd = 2, on = 1)` | quantmod `newTA`-style overlay. Plots the SuperTrend line on an existing `chartSeries` chart, colored by trend direction. |
| `chartSuperTrend(x, ..., name = deparse(substitute(x)), n = 10, multiplier = 3, atr_method = "wilder", col = c("#26a69a", "#ef5350"))` | Convenience wrapper: `chartSeries(x, name = name, ...)` followed by `addSuperTrend(...)`. |

## Dependencies

- **Imports:** `xts`, `zoo`, `TTR`, `quantmod`
- **Suggests:** `testthat (>= 3.0.0)`, `knitr`, `rmarkdown`
- **R:** `>= 3.6.0`

Rationale: TTR is the canonical R technical-analysis package and virtually every quantmod user has it installed. Reusing `TTR::ATR()` ensures our results match what users see when they call ATR directly.

## Core algorithm — `SuperTrend()`

### Inputs

- `HLC` — xts with High, Low, Close columns. Validated via `quantmod::has.HLC()`.
- `n` — ATR period, default 10 (TradingView default).
- `multiplier` — ATR band multiplier, default 3 (TradingView default).
- `atr_method` — `"wilder"` (default) | `"sma"` | `"ema"`. Passed through to `TTR::ATR(maType = ...)`.

### Algorithm

Matches TradingView's `ta.supertrend` exactly.

```
hl2  = (High + Low) / 2
atr  = TTR::ATR(HLC, n = n, maType = ...)$atr

# Basic bands
upper_basic = hl2 + multiplier * atr
lower_basic = hl2 - multiplier * atr

# Final bands — trailing: bands only loosen (move further from price),
# never tighten, unless price closes through them.
for i in 2..N:
  upper_final[i] = if (upper_basic[i] < upper_final[i-1]) || (Close[i-1] > upper_final[i-1])
                     then upper_basic[i]
                     else upper_final[i-1]
  lower_final[i] = if (lower_basic[i] > lower_final[i-1]) || (Close[i-1] < lower_final[i-1])
                     then lower_basic[i]
                     else lower_final[i-1]

# Trend direction and the SuperTrend line
trend[1] = +1   # initial trend; warm-up rows stay NA until ATR is defined
for i in 2..N:
  trend[i] = if (Close[i] > upper_final[i-1])      then +1
             else if (Close[i] < lower_final[i-1]) then -1
             else trend[i-1]

  supertrend[i] = if (trend[i] == +1) then lower_final[i] else upper_final[i]
```

### Output

An xts aligned to the input index with columns:

- `supertrend` — the indicator line value
- `trend` — `+1` (uptrend) or `-1` (downtrend)
- `upper_band` — final upper band (`upper_final`)
- `lower_band` — final lower band (`lower_final`)

The first `n` rows are `NA` (until ATR is defined), matching `TTR::ATR()`'s warm-up convention.

### Implementation note

The recursive band/trend loop cannot be vectorized without losing clarity. Implement as a plain `for` loop in R. For typical chart sizes (≤10k bars) this is fast enough. If profiling later shows it as a bottleneck, an `Rcpp` version can be dropped in without changing the API. Out of scope for v1.

### Errors

- Non-xts input → `"HLC must be an xts object"`
- Missing HLC columns → `"HLC must contain High, Low, and Close columns"`
- `n < 1` or non-integer → `"n must be a positive integer"`
- `multiplier <= 0` → `"multiplier must be positive"`
- Unknown `atr_method` → standard `match.arg()` error

## Signals — `supertrend_signals()`

### Input

The xts returned by `SuperTrend()`. Validated by checking for the `trend` column.

### Logic

Detect trend flips by comparing each row's `trend` to the previous row's.

| Event | Condition | Meaning |
|---|---|---|
| `long_entry`  | `trend[i] == +1` and `trend[i-1] == -1` | Flip from down to up — go long |
| `short_entry` | `trend[i] == -1` and `trend[i-1] == +1` | Flip from up to down — go short |
| `long_exit`   | `trend[i] == -1` and `trend[i-1] == +1` | Same bar as `short_entry` (exit any long) |
| `short_exit`  | `trend[i] == +1` and `trend[i-1] == -1` | Same bar as `long_entry` (exit any short) |

### Output

An xts aligned to the input index with four 0/1 integer columns: `long_entry`, `short_entry`, `long_exit`, `short_exit`. The first row and warm-up rows are 0 (no previous trend to compare).

`long_exit` and `short_entry` fire on the same bars by definition (a flip is one event); same for `short_exit` and `long_entry`. They are exposed as separate columns because backtesting frameworks (`quantstrat`, `PerformanceAnalytics`) typically consume entry and exit signals as distinct rules.

### Errors

- Input missing `trend` column → `"x must be the output of SuperTrend()"`

## Visualization

### `addSuperTrend(n = 10, multiplier = 3, atr_method = "wilder", col = c("#26a69a", "#ef5350"), lwd = 2, on = 1)`

A quantmod `newTA`-style overlay (`on = 1` puts it on the price panel, like `addBBands`).

**Implementation:**

1. Read the currently-charted xts via `quantmod:::get.current.chob()`.
2. Call `SuperTrend()` on it with the user's parameters.
3. Draw the line on the price panel via the chart object's draw method, mirroring how `quantmod::addBBands` is implemented.

**Color handling:** SuperTrend's signature look is the line changing color at trend flips (green when long, red when short). Plot it as **two separate line series** masked by trend direction — one with `supertrend` values where `trend == +1` and `NA` elsewhere (drawn in `col[1]`), one with `supertrend` values where `trend == -1` and `NA` elsewhere (drawn in `col[2]`). The `NA` gaps prevent connecting the two colors across flip bars. Defaults are TradingView's green/red (`#26a69a` / `#ef5350`).

### `chartSuperTrend(x, ...)`

One-liner convenience:

```r
chartSeries(x, name = name, ...)
addSuperTrend(n = n, multiplier = multiplier, atr_method = atr_method, col = col)
```

`...` passes through to `chartSeries` (theme, type, subset, TA, etc.).

### Implementation note

quantmod's `newTA` machinery uses non-standard evaluation and an internal chart-object cache (`get.current.chob`). This is the part most likely to be fiddly — mirror `addBBands`'s implementation structure closely rather than reinventing it.

### Errors

- Called with no active chart → quantmod itself errors with a clear message; no need to wrap.
- Active chart isn't OHLC → `SuperTrend()`'s own validation handles it.

## Testing

`testthat`, edition 3.

| File | Coverage |
|---|---|
| `test-supertrend.R` | Returns xts with correct columns and length; first `n` rows are NA; numeric correctness against a hand-computed fixture (~30 synthetic bars where bands and flips are verifiable by hand); each `atr_method` runs and produces different results; input-validation errors fire with expected messages; `trend` column only contains {−1, +1, NA}. |
| `test-signals.R` | Output xts has 4 integer 0/1 columns; signal counts match number of trend flips in the fixture; `long_entry` rows align with `short_exit` rows (and vice versa); validation error when input lacks `trend` column. |
| `test-addSuperTrend.R` | `chartSuperTrend()` runs without error on the built-in sample dataset, drawn to a PDF null device so no plot files are produced; `addSuperTrend()` errors gracefully when no active chart. No snapshot testing of rendered plots (fragile across R/graphics-device versions). |

**Test fixtures:** A small synthetic OHLC xts (~30 bars) constructed in `tests/testthat/helper-data.R`, with values chosen so several trend flips occur. Hand-computed expected SuperTrend values stored as a numeric vector in the helper. No external data downloads in tests (CRAN forbids network access).

## Documentation

- Roxygen2 blocks for all four exported functions, with `@return` and `@examples`. Examples that need `quantmod::getSymbols` wrapped in `\dontrun{}` (CRAN forbids network access in examples). At least one runnable example per function uses the built-in sample dataset.
- One **vignette** (`vignettes/supertrend.Rmd`): walkthrough using the built-in sample dataset — compute SuperTrend, plot it, generate signals, sketch a trivial backtest using the signal columns.
- **README.md**: install instructions, ~10-line quickstart, chart screenshot, link to vignette.

## Built-in sample data

One small dataset (`data/spy_sample.rda`, ~250 daily bars of SPY or similar OHLC). Saved as lazy data. Documented in `R/data.R` with `@source` citing where the OHLC came from. Used by examples, tests, and the vignette so none of them need network access.
